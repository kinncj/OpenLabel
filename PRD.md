# OpenLabel

## PRD

### 1. Title

**Dataset Annotation App**
Browser-only YOLO detect dataset annotator with NDJSON-first export, overlap-friendly box editing, TP/FP review states, and lossless re-import.

### 2. Context

Ultralytics supports multiple dataset/task types and many detection datasets, but the relevant scope here is **detect**: normalized xywh boxes, zero-indexed classes, overlapping objects like **person + tie**, NDJSON with a first-line dataset record plus one image record per line, and platform flows around ZIP/TAR/NDJSON upload plus NDJSON export/versioning. The supported-datasets page spans many detection taxonomies, so class presets should be an extensible registry rather than a one-off list. ([Ultralytics Docs][1])

### 3. Goals / Non-goals

**Goals**

* Upload multiple images in-browser.
* Support class presets, with **COCO-80** as the first built-in pack, plus custom classes.
* Annotate images with axis-aligned boxes.
* Support overlapping boxes as layers with explicit z-order.
* Support box review states: `TP`, `FP`, `ignore`.
* Support image review states: `complete`, `incomplete`, `negative`.
* Export a zip containing:

  * canonical `dataset.ndjson`
  * images in split folders
  * optional YOLO compatibility artifacts
  * sidecar metadata for round-trip re-import
* Re-import any dataset zip generated by the tool without losing classes, annotations, colors, z-order, or TP/FP review state.
* No backend, no server persistence, no auth.
* Next.js + React + TypeScript, Clean Architecture, SOLID, BusinessRepo layout.

**Non-goals**

* No cloud sync, collaboration, or user accounts.
* No model training or inference in v1.
* No SAM or in-browser foundation-model auto-labeling in v1.
* No segmentation, pose, OBB, or classification in v1.
* No runtime scraping of Ultralytics docs.
* No “background” or “FP” exported as a training class.

### 4. Proposal

The annotation UX should mirror the proven affordances in Ultralytics’ own editor: click-drag box creation, resize via 8 handles, center move, delete, z-order controls, undo/redo, class creation during annotation, and import of prior annotations. ([Ultralytics Docs][2])

#### 4.1 User flows

**A. Create project**

* Start from:

  * fresh image upload
  * re-imported project zip
  * raw NDJSON + images zip
* Create a local project record in IndexedDB.
* Assign default split policy: `train` required, `val` auto-created if missing.
* Manual split overrides per image and bulk split editing.

**B. Manage classes**

* Built-in class pack registry.
* Ship `COCO-80` first because Ultralytics exposes the full class map, including `person` and `tie`. ([Ultralytics Docs][1])
* Allow:

  * activate subset of preset classes
  * add custom classes
  * rename classes
  * recolor classes
  * search/filter classes
* Unified class table applies edits across dataset.

**C. Annotate**

* Layout:

  * left panel: image list/grid with split + review state + counts
  * center: annotation canvas
  * right panel: class manager + annotation stack + properties
* Core interactions:

  * draw box by dragging
  * select box
  * drag 8 handles to resize
  * drag center to move
  * delete selected box
  * multi-select boxes
  * bring forward / send backward / bring to front / send to back
  * zoom, pan, fit-to-screen
  * keyboard shortcuts for save, undo, redo, delete, z-order, class quick-select
* Overlap handling:

  * translucent fills
  * annotation stack list
  * cycle selection for stacked boxes
  * explicit z-order controls
* Box attributes:

  * class
  * normalized coordinates
  * review state: `tp | fp | ignore`
  * locked / hidden
  * z-index
  * notes (optional)

**D. Review**

* Each image has:

  * `complete`
  * `incomplete`
  * `negative`
* Export rule:

  * default is to block export for `incomplete` images
  * optional override with clear warning
* Reason:

  * partially annotated datasets can work, but Ultralytics recommends labeling all target objects and excluding unannotated images for best results. ([Ultralytics Docs][2])

**E. Export**

* Canonical export is NDJSON.
* Only `TP` boxes are exported into training annotations.
* `FP` and `ignore` remain in tool metadata only.
* If an image has zero exportable boxes, export it as a **negative image**.
* Mirror Ultralytics split behavior:

  * `train` required
  * if `val` missing and dataset has at least 2 images, auto-create a `val` subset at roughly 10% by default, while allowing manual override in UI. ([Ultralytics Docs][3])

**F. Re-import**

* Prefer `meta/project.json` for UI state.
* Prefer `dataset.ndjson` for canonical dataset content.
* Fallback path:

  * raw NDJSON + images
  * optional stretch: YOLO detect folder structure import

#### 4.2 Export contract

Because Platform NDJSON snapshots may contain URLs and annotations but not the image bytes themselves, this app should **always** include the real image files in the zip. I’m not fully certain every local-only NDJSON workflow resolves relative image files the same way when URLs are absent, so the safest interoperability choice is to ship **NDJSON + images + optional YOLO txt/YAML compatibility artifacts** together. ([Ultralytics Docs][4])

```text
dataset-annotation-app-export.zip
├── dataset.ndjson                 # canonical export
├── data.yaml                      # compatibility export
├── images/
│   ├── train/
│   ├── val/
│   └── test/                      # optional
├── labels/
│   ├── train/
│   ├── val/
│   └── test/                      # optional
└── meta/
    └── project.json               # UI-only round-trip metadata
```

**`dataset.ndjson` rules**

* First line: dataset record
* Subsequent lines: image records
* `task` must be `detect`
* `class_names` must be zero-indexed
* image records use:

  * `type: "image"`
  * `file: "<basename.ext>"`
  * `split: "train" | "val" | "test"`
  * `width`, `height`
  * `annotations: { boxes: [...] }`
* Negative image:

  * `annotations: { boxes: [] }`

**YOLO compatibility export**

* `labels/{split}/{stem}.txt`
* Empty file when image has no exported boxes
* `data.yaml` with `names`, `train`, `val`, `test` as needed

Ultralytics’ docs define detect labels as normalized `class x_center y_center width height`, allow zero-indexed classes, and say images with no objects do not require a `.txt` file. The official NDJSON converter writes an empty label file when there are no annotation lines. ([Ultralytics Docs][1])

#### 4.3 Data model

```ts
type Split = "train" | "val" | "test";
type ImageReviewState = "complete" | "incomplete" | "negative";
type BoxReviewState = "tp" | "fp" | "ignore";

type ClassDef = {
  id: number;           // zero-indexed
  name: string;
  color: string;
  source: "preset" | "custom" | "imported";
};

type BoxAnnotation = {
  id: string;
  classId: number;
  x: number;            // normalized 0..1
  y: number;            // normalized 0..1
  w: number;            // normalized 0..1
  h: number;            // normalized 0..1
  zIndex: number;
  review: BoxReviewState;
  locked: boolean;
  hidden: boolean;
  note?: string;
};

type ImageRecord = {
  id: string;
  fileName: string;
  storedBlobKey: string;
  width: number;
  height: number;
  split: Split;
  reviewState: ImageReviewState;
  annotations: BoxAnnotation[];
  hash: string;
};

type Project = {
  id: string;
  name: string;
  description?: string;
  version: number;
  createdAt: string;
  updatedAt: string;
  classes: ClassDef[];
  images: ImageRecord[];
  exportOptions: {
    includeYaml: boolean;
    includeTxtLabels: boolean;
    blockIncompleteImages: boolean;
  };
};
```

**Important rule**

* `dataset.ndjson` remains training-safe and minimal.
* `meta/project.json` stores UI-only fields such as:

  * box review state
  * z-order
  * class colors
  * image completeness
  * local hashes
  * any future editor preferences

Ultralytics describes NDJSON as extensible, but I would still keep UI-only metadata out of the training NDJSON to reduce compatibility risk. ([Ultralytics Docs][1])

#### 4.5 Definition of done

* User can upload several images and annotate them locally.
* User can select COCO classes or create new ones during annotation.
* Overlapping boxes work correctly and can be reordered.
* Box coordinates export normalized to 0..1 with zero-indexed class IDs.
* Negative images export with empty detect annotations and empty compatibility txt files.
* Re-import of an exported zip preserves classes, annotations, splits, TP/FP review state, image completeness, colors, and z-order.
* Export warns or blocks on incomplete images by default.
* Undo/redo, keyboard shortcuts, and autosave work reliably. ([Ultralytics Docs][1])

### 5. Alternatives

* **NDJSON only, no sidecar metadata**
  Rejected. It weakens round-trip fidelity for TP/FP review state, z-order, and editor metadata.

* **Encode FP as a “background” class**
  Rejected. Issue #8889 is specifically about background/negative images in multi-class detection, and the guidance is to use empty annotations for images with no objects of interest rather than inventing a special exported class. ([GitHub][5])

* **Backend upload/storage service**
  Rejected. Violates your explicit no-backend requirement.

* **Multi-task v1**
  Rejected. Detect-only is the correct scope boundary.

### 6. Trade-offs and Risks

* **Browser memory / CPU**

  * Large projects can freeze the tab during image decode or zip generation.
  * Mitigate with workers, lazy thumbnails, virtualization, and debounced autosave.

* **Partial annotation risk**

  * Mixed reviewed/unreviewed images can poison training quality.
  * Mitigate with image completeness state and export gating. ([Ultralytics Docs][2])

* **Image format mismatch**

  * Ultralytics Platform accepts more formats than browsers decode consistently.
  * v1 should prioritize JPEG, PNG, WebP, BMP, and TIFF where browser support is practical; HEIC, DNG, JP2, MPO should be non-goals or progressive enhancement. ([Ultralytics Docs][6])

* **Import trust boundary**

  * Malicious zips, path traversal, malformed NDJSON, and hostile filenames are the main threats.
  * Mitigate with:

    * zip path normalization
    * size limits
    * schema validation
    * filename sanitization
    * escaping rendered labels/classes
    * no remote auto-fetch by default

* **Data loss**

  * Browser-only apps are fragile if state is only in memory.
  * Mitigate with IndexedDB autosave + explicit export reminders.

### 7. Impact

* **Ops / cost**

  * Near-zero backend cost.
  * Static hosting only.
  * No server blast radius.

* **SRE**

  * Main failure modes move to client memory pressure, corrupted local state, and failed imports.
  * Recovery is export/re-import plus autosave snapshots.

* **Product**

  * Strong for individual annotators.
  * Weak for collaboration and large centralized data ops.

### 8. Decision

Build a **detect-only, browser-only annotation app** with **NDJSON as canonical export**, **YOLO txt/YAML as compatibility output**, and **sidecar metadata for lossless round-trip editing**.
